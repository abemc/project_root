# Phase 19 Task 3: パフォーマンス最適化 - 実装計画書

**作成日**: 2026-05-02  
**目標**: レイテンシ 50% 削減、スループット向上、Redis による L2 キャッシュ導入

---

## 1. 現状の課題
- キャッシュがメモリ内（L1）のみであり、プロセス再起動で消失する。
- 大規模な検索クエリにおいて、リトリーバーのオーバーヘッドが大きい。
- ベンチマーク環境が未整備で、定量的な性能評価が困難。

## 2. 実装コンポーネント

### 2.1 Redis による L2 キャッシュ (cache_optimizer.py)
- **役割**: 頻繁に実行される検索クエリとその結果を Redis に永続化。
- **戦略**: 
  - L1: `functools.lru_cache` (メモリ)
  - L2: Redis (キーバリューストア)
  - TTL: デフォルト 1時間（設定可能）

### 2.2 クエリ最適化 (query_optimizer.py)
- **役割**: 検索クエリのベクトル化と検索の並列化。
- **内容**: 
  - ベクトルインデックスの非同期検索。
  - 重複するサブクエリの統合。

### 2.3 インデックス戦略 (index_strategy.py)
- **役割**: 文書量に応じた最適なインデックス選択（BTREE, HASH, VECTOR）。
- **内容**: 
  - 文書追加時のインデックス自動再構築。
  - ホットスポット（頻繁に検索される領域）の検知と最適化。

### 2.4 パフォーマンスベンチマーク (performance_benchmark.py)
- **役割**: p50, p95, p99 レイテンシおよび QPS の測定。
- **内容**: `pytest-benchmark` を使用した回帰テスト。

---

## 3. 実装スケジュール

1. **Day 1-2**: Redis 環境構築 (`docker-compose` 連携) と `CacheOptimizer` 実装。
2. **Day 3-4**: `QueryOptimizer` と `IndexStrategy` による検索エンジンの強化。
3. **Day 5**: ベンチマーク測定と最終調整。

---

- ✅ キャッシュヒット率: > 85%
- ✅ 25個のパフォーマンステスト全パス
