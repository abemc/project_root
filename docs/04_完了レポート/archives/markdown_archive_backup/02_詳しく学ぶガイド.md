## A-1: LLM とは何か（5分）

### 基本定義

```
LLM = Large Language Model
    = 数十億～数兆個のパラメータを持つ
      ニューラルネットワークモデル
```

### LLMの3つの特徴

| 特徴 | 説明 | 例 |
|------|------|-----|
| **Large** | 膨大なパラメータ数 | GPT-4: 1兆パラメータ |
| **Language** | テキストを処理・生成 | 質問→回答、翻訳 |
| **Model** | 統計的パターンを学習 | 大量テキストから学習 |

### LLMが「学習」する過程

```
事前学習（Pretraining）
↓ (数ヶ月、数千GPUで学習)
├─ インターネット中の数兆トークン読む
├─ 文法・事実・推論パターンを習得
└─ 「次の単語予測」が得意に

ファインチューニング（Fine-tuning）
↓ (数時間～数日で特定タスク向け)
├─ 特定の領域データで追加学習
├─ タスク特化能力を強化
└─ 医療Q&A、法律相談などに対応
```

## A-2: LLM の内部構造（15分）

### Transformer アーキテクチャ

LLMの心臓部は **Transformer** という構造：

```
入力テキスト
  ↓
┌─────────────────────────────┐
│  Embedding Layer            │
│  （単語を数字に変換）        │
└─────────────────────────────┘
  ↓
┌─────────────────────────────┐
│  Positional Encoding        │  
│  （各単語の位置情報を追加）   │
└─────────────────────────────┘
  ↓
┌─────────────────────────────┐
│  Transformer Blocks × N     │  ← ここに複数層
│  ├─ Self-Attention Layer    │
│  ├─ Feed-Forward Layer      │
│  └─ Normalization           │
│  （N = 10～100層程度）       │
└─────────────────────────────┘
  ↓
┌─────────────────────────────┐
│  Output Projection          │
│  （数字をテキストに変換）    │
└─────────────────────────────┘
  ↓
出力テキスト
```

### Self-Attention メカニズム（最重要！）

**Self-Attention とは**: 入力文の各単語が、他のどの単語に注目すべきか自動判定する機構

**具体例：**

```
入力文: 「銀行は新しい支店を開きました」

"銀行"は何に注目すべき?
├─ "新しい" → 100% (銀行の属性を説明)
├─ "支店" → 90% (銀行の一部)
└─ "開きました" → 70% (銀行のアクション)

"支店"は何に注目すべき?
├─ "銀行" → 95% (支店の所属)
├─ "新しい" → 80% (支店の属性)
└─ "開きました" → 60% (支店のアクション)
```

**計算流れ：**

```python
# 簡略版
Query = 現在の単語が「何を知りたいか」
Key = 他の単語が「何を持っているか」の特徴
Value = 他の単語の「実際の値」

Attention = softmax(Query · Key) · Value
        = 関連度 × 他の単語の値
        = 加重平均
```

### Multi-Head Attention

複数の異なる質問を同時に実行：

```
入力: 「医療用ロボットの安全基準」

Head 1: 「医療」に注目 → Medical domain特徴抽出
Head 2: 「ロボット」に注目 → Technical domain特徴抽出  
Head 3: 「安全基準」に注目 → Legal domain特徴抽出
Head 4: 「基準」に注目 → Business/管理面抽出

合計: 4つの視点で同時分析
   = より豊かな理解
```

## A-3: LLM の推論過程（10分）

### トークンとは

```
テキスト → トークン化 → 数字に変換

例：
"医療用ロボットの安全基準"
  ↓ トークン化
["医療", "用", "ロボット", "の", "安全", "基準"]
  ↓ 数字化
[4521, 203, 8832, 19, 3042, 5501]
  ↓ Embedding
[[0.2, -0.5, 0.8, ...], [0.1, 0.3, -0.2, ...], ...]
  (768次元のベクトル × 6個)
```

### LLM の生成過程

```
質問: "医療用ロボットの安全基準について説明してください"

Step 1: 質問をトークン化
[質問トークン...]

Step 2: Transformerで処理
Attention × 48層 = 深い理解

Step 3: 次の単語を確率計算
P(次は"医療") = 0.02
P(次は"ロボット") = 0.001
P(次は"安全") = 0.85  ← 選択!

Step 4: 選んだトークンを追加
[質問トークン... 安全]

Step 5: 繰り返し
[質問トークン... 安全 基準]
[質問トークン... 安全 基準 は]
...
完成: "安全基準は医療現場で極めて重要です..."
```

## A-5: LLM の学習メカニズム - 自薦学習とファインチューニング（20分）

### なぜLLMの学習を理解する必要があるのか

LLMの性能は、その「学習プロセス」で決まります。同じアーキテクチャでも、学習方法次第で結果は大きく変わります。特に、自薦学習（Self-Supervised Learning）とファインチューニング（Fine-tuning）は、LLMが高性能を実現するための2つの重要な段階です。

### 自薦学習（Self-Supervised Learning）とは

#### 基本概念

**自薦学習** = 「正解ラベルなしで、データ自体から学習パターンを抽出する学習方法」

```
従来の機械学習の問題:
  正解ラベル（アノテーション）を用意する
    ↓
  「100万データには、100万の正解ラベルが必要」
    ↓
  コストが莫大！ → スケーラビリティの問題

自薦学習の力:
  正解ラベルなしに「データ自体の構造」から学習
    ↓
  インターネット上の数兆トークンを直接利用
    ↓
  低コストで大規模学習が可能 ✅
```

#### 自薦学習の具体例：マスキング言語モデル訓練

**シンプルな例で理解しよう：**

```
元の文: "医療用ロボットは患者の治療を支援します"

Step 1: 単語をマスク（隠す）
        "医療用ロボットは患者の[MASK]を支援します"
        "医療用[MASK]は患者の治療を支援します"

Step 2: モデルに予測させる
        [MASK]に何が入るか？
        ├─ 1番目: 「治療」 → 正解!
        └─ 2番目: 「ロボット」 → 正解!

Step 3: 自動的に損失を計算
        予測値 vs 実際の値
        → 差を減らすように学習

Step 4: 繰り返し
        数兆トークンで何度もこれを繰り返す
        → モデルが言語パターンを習得
```

#### 自薦学習で何が学習されるのか

```python
# LLMが学習する知識体系（自薦学習の結果）

学習内容 = {
    "言語構造": {
        "文법": "主語 + 述語 + 目的語",
        "時制": "過去形・現在形・未来形の使い分け",
        "修飾関係": "形容詞が名詞を修飾する関係"
    },
    
    "世界知識": {
        "事実知識": "パリはフランスの首都",
        "常識知識": "バナナは皮を剥いて食べる",
        "因果関係": "雨が降る → 地面が濡れる",
        "科学知識": "H2O = 水分子"
    },

        "論理推論": "AはBである、BはCである → AはCである",
        "必然性": "火傷した → 痛い",
        "確率的推論": "曇り空 → 雨が降る確率は高い"
    },
    
    "表現能力": {
        "感情表現": "幸せ・悲しい・怒りの表現方法",
        "様式": "フォーマル・カジュアル・専門的な表現",

    }

```

#### 実装のポイント：大規模言語モデルの事前学習

```python
class SelfSupervisedPretraining:
    def __init__(self, model, data_loader):
        self.model = model
        self.data_loader = data_loader
    
    def pretrain(self, num_epochs=3):
        """事前学習の実装"""
        for epoch in range(num_epochs):
            total_loss = 0
            
            for batch in self.data_loader:
                # Step 1: テキストをトークン化
                tokens = tokenize(batch)
                
                # Step 2: ランダムに単語をマスク（15%）
                masked_tokens, mask_labels = mask_tokens(
                    tokens, 
                    mask_ratio=0.15
                )
                
                # Step 3: マスクされた単語を予測
                predictions = self.model(masked_tokens)
                
                # Step 4: 損失を計算
                loss = self.calculate_loss(
                    predictions,
                    mask_labels
                )
                
                # Step 5: 逆伝播で重みを更新
                loss.backward()
                self.optimizer.step()
                
                total_loss += loss.item()
            
            print(f"Epoch {epoch+1}: Loss = {total_loss/len(data_loader)}")
            
        return self.model
```

#### 自薦学習の効果

```
Pretraining前: ランダムな回答
  Q: "医療用ロボットの用途は？"
  A: "●■◆□◇●◆" (ランダムなトークン)

Pretraining後: ある程度の知識
  Q: "医療用ロボットの用途は？"
  A: "医療用ロボットは[医療分野/患者の治療/手術支援]に使用されます"
    ↑ 正確ではないかもしれないけど、言語パターンを習得
```

### ファインチューニング（Fine-tuning）とは

#### 基本概念

**ファインチューニング** = 「特定のタスクや領域に特化させるための追加学習」

```
Pretraining の結果:
  ├─ 一般的な言語知識 ✅
  ├─ 基本的な常識 ✅
  └─ 幅広い分野の知識 ✅
  
しかし:
  └─ 特定の領域では精度が低い ❌
     例: 医療専門用語、法律用語など

Fine-tuning の役割:
  Pretraining で学んだ知識をベースに
    ↓
  特定領域のデータで追加学習
    ↓
  その領域での高精度が実現される ✅
```

#### ファインチューニングの3つのアプローチ

**アプローチ1：教師あり学習によるファインチューニング**

```
データ形式:
  入力: "医療用ロボットの安全基準を説明してください"
  正解: "医療用ロボットは以下の基準に従う必要があります..."

プロセス:
  Step 1: 事前学習済みモデルをロード
  Step 2: 特定領域（医療）のデータで学習
  Step 3: 「正解との差」を最小化するよう調整
  Step 4: 医療特化モデルが完成

効果:
  Pretraining: 一般的な医療知識 (正確度 70%)
        ↓
  Fine-tuning: 医療専門知識 (正確度 95%)
```

**アプローチ2：適応的ファインチューニング（LoRA）**

```
課題：
  事前学習済みモデルの全パラメータを更新すると
  → メモリが膨大に必要 (数GB～数TB)
  → 計算コストが莫大

解決策：Low-Rank Adaptation (LoRA):

  元のパラメータ
  ├─ 固定（変更しない）← 事前学習の知識を保持
  └─ 新しい小さなアダプタを追加
     ├─ サイズが千倍以上小さい
     ├─ メモリ効率的
     └─ 大幅な性能向上

効果:
  通常のFinetuning: 7B parameters 更新 → 28GB メモリ必要
  LoRA: 0.01B parameters 更新 → 500MB メモリで十分
```

**アプローチ3：マルチタスク・ファインチューニング**

```
複数領域での学習:

データセット構成:
  ├─ 医療データ: 100万トークン
  ├─ 法律データ: 100万トークン
  ├─ 技術データ: 100万トークン
  └─ ビジネスデータ: 100万トークン

学習方法:
  Step 1: 全領域のデータを混合
  Step 2: モデルが「どの領域か」を判定できるようにマーク
  Step 3: 各領域のタスクを同時に学習
  Step 4: マルチドメイン対応の高性能モデルが完成

利点:
  ├─ 単一領域より高性能（ドメイン知識の相乗効果）
  ├─ 領域横断的な推論が可能
  └─ Phase 7 システムの核となる能力
```

#### ファインチューニングの実装例

```python
class DomainFineTuning:
    def __init__(self, pretrained_model, domain="medical"):
        self.model = pretrained_model
        self.domain = domain
        self.domain_adapter = LoRA(
            base_model=self.model,
            rank=8,  # LoRA の秩
            alpha=16  # スケーリング係数
        )
    
    def finetune_on_domain_data(self, 
                                 domain_dataset,
                                 num_epochs=3,
                                 batch_size=32):
        """特定領域でのファインチューニング"""
        
        dataloader = DataLoader(
            domain_dataset,
            batch_size=batch_size,
            shuffle=True
        )
        
        optimizer = AdamW(
            self.domain_adapter.parameters(),
            lr=2e-4  # ファインチューニングは低い学習率
        )
        
        for epoch in range(num_epochs):
            total_loss = 0
            
            for batch_idx, batch in enumerate(dataloader):
                # Step 1: 入力と正解を準備
                inputs = batch['input_ids'].to(self.device)
                labels = batch['labels'].to(self.device)
                
                # Step 2: モデルの推論
                outputs = self.model(inputs)
                predictions = outputs.logits
                
                # Step 3: 損失を計算
                loss = F.cross_entropy(predictions, labels)
                
                # Step 4: 逆伝播
                optimizer.zero_grad()
                loss.backward()
                
                # Step 5: 勾配をクリッピング（安定性向上）
                torch.nn.utils.clip_grad_norm_(
                    self.domain_adapter.parameters(),
                    max_norm=1.0
                )
                
                # Step 6: パラメータ更新
                optimizer.step()
                
                total_loss += loss.item()
                
                # 進捗表示
                if (batch_idx + 1) % 100 == 0:
                    avg_loss = total_loss / (batch_idx + 1)
                    print(f"Epoch {epoch+1}, Batch {batch_idx+1}: "
                          f"Loss = {avg_loss:.4f}")
            
            print(f"Epoch {epoch+1} 完了: 最終ロス = {total_loss/len(dataloader):.4f}")
        
        return self.model
```

#### ファインチューニングの効果測定

```
医療タスクでの精度比較：

タスク: 医学論文から関連情報を抽出

  事前学習のみ (Pretraining):

    ├─ 因果関係の抽出: 55%
    ├─ 信頼度評価: 45%
    └─ 総合精度: 53%

  ファインチューニング後 (Medical Fine-tuned):
    ├─ 医学用語の理解: 94%

#### ファインチューニングの落とし穴と対策

| 問題 | 原因 | 対策 |
|------|------|------|
| **破局的忘却（Catastrophic Forgetting）** | 事前学習の知識を過剰に上書き | 学習率を低く、エポック数を少なく |
| **過学習（Overfitting）** | 訓練データに過度に適応 | 検証セットでモニタリング、早期停止 |
| **ドメインシフト** | 訓練データと実運用データが異なる | テストセットで常に検証 |
| **計算リソース不足** | メモリ・時間が想定以上必要 | LoRA や勾配チェックポイント使用 |

   ├─ 時間: 数ヶ月、数千GPU
   ├─ コスト: 非常に高額 (数百万ドル)
   ├─ 結果: 一般的な大規模言語モデル
   └─ 例: GPT-4の事前学習
   
        ↓ (モデルをダウンロード)

   ├─ 目的: 特定領域への特化

   ├─ 時間: 数時間～数日、単一マシン
   ├─ コスト: 低〜中程度 (数千ドル)
   ├─ 結果: 領域特化型の高性能モデル
   └─ 例: 医療用のGPT-4、法律用のGPT-4
   
        ↓ (完成・デプロイ)

3. 本番運用
   ├─ 高精度・低レイテンシ・信頼できる回答

# PART B 🔧: 3つの推論エンジン完全解説

## B-1: KnowledgeIntegrator（知識統合エンジン）

### 役割

```
複数のドメイン知識を統合して、
より完全で正確な回答を作成する
```

### 具体例：医療用ロボット

```
ユーザーの質問: "医療用ロボットの承認取得に必要なステップは？"

このユーザーの質問は3つのドメインを含む：

┌─────────────────┐
│ Medical ドメイン │
│ ├─ 医療デバイス規格
│ ├─ 患者安全
│ └─ 臨床試験
└─────────────────┘
        ↓ 統合
┌─────────────────┐
│ Legal ドメイン  │
│ ├─ 規制要件
│ ├─ 認可プロセス
│ └─ 弁護士相談
└─────────────────┘
        ↓ 統合
┌─────────────────┐
│ Technical ドメイン│
│ ├─ 装置の仕様
│ ├─ ソフトウェア検証
│ └─ セーフティシステム
└─────────────────┘

        results = {}

            confidence=confidence,
            sources=results
        )

## B-2: CausalReasoningEngine（因果推論エンジン）

### 役割

```
「Aが起きたらなぜBが起きるのか」を理解する
因果関係を推論する
```

### 具体例：医学的推論

```
質問: 「患者の血圧が上昇した理由は？」

CausalReasoningEngine の推論フロー:

① 直接原因を検索
├─ 最近のストレス? → 因果度 0.7
├─ 塩分過剰摂取? → 因果度 0.6
└─ 運動不足? → 因果度 0.5

② 間接原因を検索
├─ ストレス ← 仕事の変化
├─ 塩分 ← 食習慣変化
└─ 運動 ← 足の負傷

③ 因果チェーン構築
仕事変化 → ストレス → 血圧上昇
足負傷 → 運動不足 → 血圧上昇

④ 最強因果を特定
「ストレスが主要な因果因子」と判定
（信頼度 0.78）

出力:
結論: 仕事ストレスが主原因
理由: ストレス → ホルモン変化 → 血管収縮
対策: ストレス管理、瞑想、休息
```

### バックワード推論（重要！）

```
通常の推論（順向き）:
原因 (A) → 結果 (B)
「ストレス」→ 「血圧上昇」

バックワード推論（逆向き）:
結果 (B) ← 原因 (A)
「血圧上昇が観察された」← 「何が原因？」

これの何が重要か:
- 複雑なシステムで「なぜ」を理解
- 複数の因果チェーンを逆追跡
- 最も可能性の高い原因に到達

医療例:
症状 ← 検査値の異常 ← 臓器機能低下 ← 病気
←バックワード推論で「病名」を特定
```

### 実装概要

```python
class CausalReasoningEngine:
    def reason_causality(self, observation, domain):
        # Step 1: 知られた因果ルール取得
        causal_rules = self.load_domain_rules(domain)
        
        # Step 2: 各ルール評価
        candidates = []
        for rule in causal_rules:
            if rule.matches(observation):
                score = rule.calculate_likelihood()
                candidates.append((rule, score))
        
        # Step 3: スコアでソート
        candidates.sort(key=lambda x: x[1], reverse=True)
        
        # Step 4: 因果チェーン構築
        causal_chain = self.build_chain(candidates[0])
        
        # Step 5: 検証
        verified = self.verify_chain(causal_chain)
        
        return CausalReasoning(
            primary_cause=candidates[0][0],
            chain=verified,
            confidence=candidates[0][1]
        )
```

### 確実性レベルの定義

```
信頼度スコア (0.0 ～ 1.0)

1.0  ■■■■■■■■■■ 完全に確実
     └─ 医学常識、物理法則

0.8  ■■■■■■■■   非常に可能性が高い

     └─ 主要な情報源で確認

0.4  ■■■■       低めの確実性
     └─ 単一情報源のみ

0.2  ■■         疑わしい
     └─ 矛盾した情報がある

0.0  ■          推測段階
     └─ データ不足、確実性ゼロ
```

### 信頼度の計算例

```
質問: 「人工膝関節の耐用年数は？」

複数の情報源から評価:

┌──────────────────────────────┐
│① 医学文献                     │
│   「通常15-20年」             │
│   信頼度スコア: 0.95          │
│   理由: ピアレビュー済み      │
└──────────────────────────────┘
                ↓
┌──────────────────────────────┐
│② メーカー公式スペック         │
│   「設計寿命20年」            │
│   信頼度スコア: 0.85          │
│   理由: 公式だが保証ではない  │
└──────────────────────────────┘
                ↓
┌──────────────────────────────┐
│③ 医者の臨床経験               │
│   「実際は12-18年見てる」     │
│   信頼度スコア: 0.75          │
│   理由: サンプル数限定        │
└──────────────────────────────┘

統合計算:
最終スコア = 
  0.95 × 0.4 (文献の重み)
+ 0.85 × 0.3 (公式の重み)
+ 0.75 × 0.3 (経験の重み)
= 0.87

結論: 信頼度 87% で「15-20年」が正解
```

```

          → 返答保留

# PART C 🗂️: ドメイン知識の実装詳細

## C-1: ドメインとは何か

### ドメインの本質

**ドメイン（Domain）** とは、簡単に言うと「**専門知識の領域**」のことです。世界中のあらゆる知識は、複数の異なる領域に分けることができます。例えば、医者が使う医学知識は「医療ドメイン」、弁護士が使う法律知識は「法律ドメイン」というように、それぞれが独立した知識体系を持っています。

**なぜドメインで分ける必要があるのか？**

異なる領域の知識には、異なるルール・常識・判断基準があるからです。同じテーマでも、どのドメインから見るかで答えが変わります：

**例：「医療用ロボットの販売」という話題**

| ドメイン | 視点 | 重視する点 |
|---------|------|----------|
| **医療** | 臨床的視点 | 患者の安全性、治療効果 |
| **法律** | 法的視点 | 法的承認、規制要件 |
| **ビジネス** | 商業的視点 | 市場需要、利益率 |
| **技術** | 工学的視点 | 実装可能性、パフォーマンス |
| **科学** | 研究的視点 | 科学的根拠、検証可能性 |

同じロボットについて質問しても、どのドメインの知識を適用するかで答えが大きく異なります。医療ドメインなら「患者にとって本当に安全か」を重視し、ビジネスドメインなら「どのくらい儲かるか」を重視するのです。

### Phase 7 のドメインシステム

Phase 7の推論エンジンは、ユーザーが入力した質問から必要なドメイン知識を自動判定し、複数のドメイン知識を統合することで、より完全で正確な回答を生み出します。

**処理フロー：**

```
ユーザーの質問入力
  ↓
「どのドメインの知識が必要か」を自動判定
  ↓
該当するドメイン知識を引き出す
  ↓
複数ドメイン知識を組み合わせる
  ↓
矛盾を解決し、ドメイン固有の制約を考慮
  ↓
統合されたアンサーを生成
```

例えば、「医療用マイクロロボットの特許戦略」という質問がユーザーから来たら：
1. 医療ドメイン知識 → 医療デバイスの特性
2. 法律ドメイン知識 → 特許法と知的財産保護
3. 技術ドメイン知識 → ロボット技術の仕様
4. ビジネスドメイン知識 → 市場戦略

これら4つの異なるドメイン知識を統合して、初めて「完全な回答」が得られるのです。

2. **説明可能性**
   - どのドメイン視点から答えているかが明確
   - 「医療的には○○、法的には××」と区別して説明可能

3. **信頼度評価**
   - ドメイン間で信頼度が異なることを管理
   - 医学文献は信頼度高く、ニュースは信頼度低い、など区別

医師は、これら4つのドメイン情報を統合して、初めて完全な判断ができるのです。

def infer_domain(query):
    # Step 1: キーワードマッチング
    keyword_scores = {}
    for domain, keywords in DOMAINS.items():
        score = count_matches(query, keywords)
        keyword_scores[domain] = score
    
    # Step 2: ドメイン間の関係性を考慮
    adjusted_scores = apply_relationships(keyword_scores)
    
    # Step 3: コンテキスト分析
    context_domain = analyze_context(query)
    adjusted_scores[context_domain] *= 1.2  # ブースト
    
    # Step 4: 最高スコアを選択
    primary_domain = max(adjusted_scores, 
                         key=adjusted_scores.get)
    confidence = adjusted_scores[primary_domain]
    
    return primary_domain, confidence
```

## D-3: 実装コード例

### 完全な処理フロー（Python）

```python
class Phase7ReasoningPipeline:
    """Phase 7 システムの完全推論パイプライン"""
    
    def __init__(self):
        self.domain_manager = DomainKnowledgeManager()
        self.knowledge_integrator = KnowledgeIntegrator()
        self.causal_reasoner = CausalReasoningEngine()
        self.uncertainty_mgr = UncertaintyManager()
        self.rag_engine = Phase7KnowledgeIntegrationEngine()
        
    def process_query(self, query: str) -> dict:
        """ユーザーの質問から回答まで"""
        
        # Step 1: ドメイン推定
        domain, domain_conf = self.domain_manager\
            .infer_domain_from_query(query)
        print(f"[1] ドメイン判定: {domain} ({domain_conf:.1%})")
        
        # Step 2: 関連ドメイン検出
        related_domains = self.domain_manager\
            .find_related_domains(domain, query)
        print(f"[2] 関連ドメイン: {related_domains}")
        
        # Step 3: RAG 検索
        retrieved_docs = self.rag_engine\
            .retrieve_relevant_docs(query, domain)
        print(f"[3] 検索結果: {len(retrieved_docs)}件")
        
        # Step 4: 知識統合
        integrated = self.knowledge_integrator\
            .integrate_knowledge(
                primary_domain=domain,
                related_domains=related_domains,
                query=query,
                retrieved_docs=retrieved_docs
            )
        print(f"[4] 知識統合完了")
        
        # Step 5: 因果推論
        causality = self.causal_reasoner\
            .reason(query, integrated)
        print(f"[5] 因果推論実行")
        
        # Step 6: 不確実性評価
        confidence = self.uncertainty_mgr\
            .evaluate_confidence(
                query=query,
                domain=domain,
                retrieved_docs=retrieved_docs,
                causality=causality
            )
        print(f"[6] 信頼度評価: {confidence:.1%}")
        
        return {
            "domain": domain,
            "related_domains": related_domains,
            "answer": integrated.result,
            "causality": causality.chain,
            "confidence": confidence,
            "sources": integrated.sources,
            "retrieved_count": len(retrieved_docs)
        }
```
