## 🎯 概要

### 目的

**24/7 連続監視体制での迅速なインシデント対応**

- Critical 重大度: **5 分以内対応**
- High 重大度: **15 分以内対応**
- Medium 重大度: **1 時間以内対応**

### オンコール体制

**ローテーション:**
```
24h × 7日 × 3班体制 (各班 8h シフト)

08:00-16:00: Day Shift (運用チーム)
16:00-00:00: Evening Shift (SOC チーム)
00:00-08:00: Night Shift (開発チーム)

土日祝祭日:
  08:00-16:00: Standby (On-call 1)
  16:00-24:00: Standby (On-call 2)
  00:00-08:00: Standby (On-call 3)
```

## ⚡ 初動対応フロー

```
アラート検出
  ↓ (0-5 秒)
【自動通知】
  Slack, Email, SMS, Phone (優先度順)
  ↓ (5-10 秒)
【通知確認】
  受信確認: "Acknowledged" をクリック
  ↓ (10-30 秒)
【情報確認】
  ダッシュボード・ログで詳細確認
  ↓ (30-60 秒)
【初動判断】
  - 誤検知 → Close
  - 軽微 → 対応予定
  - 中程度 → 対応開始
  - 重大 → 即座エスカレーション
  ↓ (1-5 分)
【対応実施】
  - 自動復旧可能 → 実施
  - 手動対応要 → 開始
  - 外部リソース要 → 要請
  ↓
【経営層報告】
  Critical/High: 15 分ごと更新
  ↓
【復旧・検証】
  ↓
【事後報告】
  ポストモーテム: 翌営業日
```

## 🚨 重大度別対応手順

### 🔴 Critical (重大度: 赤)

**対応時間: 5 分以内**

#### 事例
- 本番システム停止
- 認証システム完全ダウン
- 全リージョン故障
- セキュリティブリーチ検知
- SLA 99.99% 喪失

#### 初動手順

```
1. 通知確認 (即座)
   [ ] Slack/Email/SMS 確認
   [ ] "Acknowledged" ボタンクリック
   [ ] 詳細確認開始

2. 状況把握 (1 分以内)
   [ ] Grafana ダッシュボード表示
   [ ] Prometheus アラート確認
   [ ] ログイン可否確認
   [ ] 影響範囲推定

3. チーム招集 (1 分以内)
   [ ] PagerDuty で次シフト通知
   [ ] Slack #critical-incidents チャネルに投稿
   [ ] 技術リーダーに電話通知
   [ ] 管理層に速報メール

4. 対応判断 (2 分以内)
   [ ] 対応マニュアル該当箇所確認
   [ ] 自動復旧実行 vs 手動対応判定
   [ ] 必要なら外部ベンダー通知

5. 対応実施 (5 分以内)
   [ ] 根本原因特定
   [ ] 応急処置実施
   [ ] 影響最小化対応
   [ ] ステータス更新 (1 分ごと)

6. 報告 (継続)
   [ ] 15 分ごとに経営層へ更新
   [ ] Slack での進捗報告 (5 分ごと)
```

#### 実行例

```bash
# Step 1: ダッシュボード確認
open https://grafana.internal/d/esp-7layer-monitoring

# Step 2: チーム通知
curl -X POST https://hooks.slack.com/services/YOUR/WEBHOOK \
  -H 'Content-Type: application/json' \
  -d '{
    "channel": "#critical-incidents",
    "text": "🔴 CRITICAL: 本番認証システム停止",
    "attachments": [{
      "color": "danger",
      "fields": [
        {"title": "Severity", "value": "CRITICAL", "short": true},
        {"title": "Component", "value": "Authentication Engine", "short": true},
        {"title": "On-Call", "value": "Alice", "short": true},
        {"title": "Start Time", "value": "2026-04-18 13:45:00", "short": true}
      ]
    }]
  }'

# Step 3: ログ確認
kubectl logs -n production deployment/authentication-engine \
  --tail=100 | grep -i error

# Step 4: 自動復旧実行
python src/phase10/auto_recovery.py \
  --component authentication \
  --action restart \
  --notify-team true
```

### 🟠 High (重大度: 橙)

**対応時間: 15 分以内**

#### 事例
- 認証遅延 (> 500ms)
- SOC イベント処理バックログ
- 単一リージョン故障
- インシデント自動対応失敗

#### 初動手順

```
1. 通知確認 (即座)
   [ ] Slack/Email 確認
   [ ] "Acknowledged" クリック

2. 詳細確認 (2 分以内)
   [ ] メトリクス確認
   [ ] 影響範囲推定
   [ ] トレンド分析 (過去1時間)

3. チーム準備 (3 分以内)
   [ ] Slack 投稿 (#incidents)
   [ ] 技術チーム通知
   [ ] 予備担当者スタンバイ

4. 対応方法決定 (5 分以内)
   [ ] 自動復旧実施可否判定
   [ ] 手動対応手順確認
   [ ] リソース確保

5. 対応実施 (15 分以内)
   [ ] 対応開始
   [ ] 進捗報告 (10 分ごと)

6. フォローアップ
   [ ] 30 分後に状態確認
   [ ] 解決確認後、クローズ
```

---

### 🟡 Medium (重大度: 黄)

**対応時間: 1 時間以内**

#### 事例
- 脅威検知精度低下
- ディスク容量警告
- API ゲートウェイ個別エラー
- セキュリティ機能部分的低下

#### 初動手順

```
1. 通知確認
   [ ] Slack #alerts チャネル確認

2. 優先度判定 (5 分)
   [ ] 業務影響度の有無
   [ ] 解決の難易度
   [ ] 他のインシデントとの競合

3. 対応予定設定
   [ ] 対応予定時刻を記録
   [ ] 他のタスクとの調整
   [ ] チーム内で共有

4. 対応実施 (1 時間以内)
   [ ] 根本原因特定
   [ ] 対応実施
   [ ] 検証完了

5. クローズ
   [ ] ステータス更新
   [ ] チーム通知
```

---

### 🟢 Info (通知: 緑)

**対応時間: 営業日内**

#### 事例
- 通常のログイン成功イベント
- 定期メンテナンス通知
- 監査情報

#### 初動手順

```
1. ログ記録
   [ ] Slack に記録

2. 必要に応じて対応
   [ ] 統計情報収集
   [ ] トレンド分析
```

## 📋 事後対応

### ポストモーテム実施

**実施時期:** インシデント発生翌営業日 14:00

**参加者:**
- On-call 担当者
- 技術リーダー
- 関連チーム
- 管理層 (必要に応じて)

**議題:**

```
1. タイムライン確認 (5 分)
   - いつ検出されたか
   - いつ対応が始まったか
   - いつ解決したか

2. 根本原因分析 (15 分)
   - なぜ起きたのか
   - 根本原因は何か
   - どうやって判明したか

3. 影響分析 (5 分)
   - ユーザーへの影響
   - ビジネスへの影響
   - データ喪失の有無

4. 改善アクション (20 分)
   - 短期 (今週): 臨時対応
   - 中期 (1-4 週): 根本修正
   - 長期 (1-3 月): システム改善

5. 学習・共有 (5 分)
   - チーム全体への情報共有
   - ドキュメント更新
   - トレーニング必要性
```
