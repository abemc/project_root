## 🎯 概要

このドキュメントは、開発チームから運用・サポート・保守チームへの **技術ナレッジの体系的な移行** を支援します。

### 対象者

| チーム | 人数 | 主要責任 | 必須知識 |
|-------|------|--------|---------|
| **運用チーム** | 3人 | 日次監視・アラート対応 | インフラ・監視・SLA |
| **サポートチーム** | 2人 | 顧客サポート・問題解決 | ユーザー機能・トラブルシューティング |
| **保守チーム** | 2人 | バグ修正・小規模機能追加 | コード・テスト・デプロイ |

### よくある問い合わせ対応例

**Q: ログインできません**

```
対応ステップ:
1. ユーザー名確認 (正しく入力されているか)
2. パスワードリセットリンク送付
3. ブラウザキャッシュクリア指示
4. 別ブラウザで試行指示
5. それでもダメなら: 管理画面でユーザー確認 → reset
```

**Q: MFA コードが届きません**

```
対応ステップ:
1. メールアドレス確認
2. メールスパムフォルダ確認指示
3. 時刻同期確認 (デバイスの時刻)
4. TOTP リセット申請
5. バックアップコード使用
```

## ❓ FAQ

### よくある質問

**Q: 本番システムへのアクセスはいつから?**

```
A: セキュリティトレーニング + 認定試験合格後
   通常: Week 4 以降
   → Vault シークレット・SSH キー配付
```

**Q: アラートが多い場合の対応は?**

```
A: 閾値調整フロー:
   1. 5日分のデータ収集
   2. 統計分析 (平均・標準偏差)
   3. 新しい閾値 = 平均 + 3σ
   4. テックリーダーレビュー
   5. 本番反映
```

**Q: チケット優先度の判断基準は?**

```
A: 
   P1 (Critical): ユーザーに影響・24h対応
   P2 (High): 複数ユーザー影響・3日対応
   P3 (Medium): 1ユーザー影響・1週間対応
   P4 (Low): バグなし・改善提案・好意
```
