## 🚨 ロールバック手順

### 緊急ロールバック (< 5分)

**トリガー条件**:
- エラー率 > 1.0%
- 平均レイテンシ > 5000ms
- システムメモリ使用率 > 95%
- コンポーネントのクラッシュ検出

**自動実行ステップ**:

```bash
# ステップ1: 新システムへのトラフィック遮断 (~10秒)
aws elb set-instance-health \
  --load-balancer-name prod-elb \
  --instances i-new-instance-id \
  --state OutOfService

# ステップ2: 旧システムへ100%トラフィック復帰 (~10秒)
aws elb set-instance-health \
  --load-balancer-name prod-elb \
  --instances i-old-instance-id \
  --state InService

# ステップ3: 状態確認
curl -s http://api.example.com/health | jq .

# 期待値:
# {
#   "status": "healthy",
#   "version": "v1",
#   "uptime": 123456
# }

# ステップ4: チーム報告
echo "ROLLBACK COMPLETED - Phase X" | \
  curl -X POST -d @- https://hooks.slack.com/services/YOUR/WEBHOOK/URL
```

**復帰時間**: < 30秒 + 確認 5分 = **合計 < 5分38秒**

### 手動ロールバック (5-30分)

**判断条件**:
- 軽微なバグで修正可能性あり
- パフォーマンス低下が一時的
- データ整合性は保持

**ステップ**:

1. **トラフィック配分を50%に戻す**
   ```bash
   aws elb set-instance-weights \
     --load-balancer-name prod-elb \
     --instances \
       i-new-instance-id=50 \
       i-old-instance-id=50
   ```
   
   所要時間: ~5分

2. **メトリクス監視 (15分)**
   ```
   監視項目:
   - エラー率が0.5%以下に改善
   - レイテンシが500ms以下に改善
   - CPU/メモリが正常値に復帰
   ```

3. **詳細分析 (5分)**
   ```bash
   # ログ確認
   tail -1000 /var/log/rag_engine.log | grep ERROR
   
   # エラー内容を分析して対応方針を決定
   ```

4. **対応方針決定**
   ```
   修正可能: デプロイ修正版を新インスタンスに適用
   → その後 Phase 2から再試行
   
   不可: 完全ロールバック
   → 旧システムへ100% 復帰
   → デプロイメント延期
   ```
