## 📅 日常運用手順

### 朝礼 (08:00-08:30)

```bash
#!/bin/bash
# scripts/daily_standup.sh

echo "=== 朝礼レポート ==="
echo "日付: $(date '+%Y-%m-%d')"
echo ""

# 1. 前夜のインシデント
echo "【インシデント】"
curl -s http://monitoring:9093/api/v1/alerts | \
  jq '.data[] | select(.status=="firing") | {severity, summary}' | \
  wc -l | xargs echo "  Active Alerts:"

# 2. メトリクス確認
echo ""
echo "【システムメトリクス】"
echo "  認証成功率: $(curl -s http://monitoring:9090/api/v1/query?query=auth_success_rate | jq '.data.result[0].value[1]' | tr -d '"')%"
echo "  SOC 処理遅延: $(curl -s http://monitoring:9090/api/v1/query?query=soc_latency_p99 | jq '.data.result[0].value[1]' | tr -d '"')ms"
echo "  CPU 使用率: $(kubectl top nodes | awk 'NR==2 {print $3}')"

# 3. リソース確認
echo ""
echo "【リソース】"
kubectl describe nodes | grep -E "Allocated|Remaining"
```

**実行例:**
```

  CPU 使用率: 45%

【リソース】
  CPU Allocated: 24/48 (50%)

```

### 定期チェック (毎2時間)

```bash
#!/bin/bash
# check_system_health.sh

echo "[$(date)] === システムヘルスチェック ==="

# 1. K8s ノード確認
echo ""
echo "【Kubernetes Nodes】"
kubectl get nodes -o wide

# 2. Pod 状況
echo ""
echo "【Pod Status】"
kubectl get pods -n production --no-headers | \
  grep -E "CrashLoop|Error|Pending" || echo "  ✓ All pods healthy"

# 3. メトリクス
echo ""
echo "【CPU/Memory】"
kubectl top nodes

# 4. DB チェック
echo ""
echo "【Database】"
psql -h db-primary.internal -c "SELECT 'DB OK' as status;" 2>&1

# 5. キャッシュ確認
echo ""
echo "【Cache (Redis)】"
redis-cli -h cache.internal PING

# 6. アラート
echo ""
echo "【Alerts】"
curl -s http://monitoring:9093/api/v1/alerts | jq '.data | length'
```

## 🔔 監視・アラート

### 主要メトリクス (SLA 目標)

| レイヤー | メトリクス | 目標値 | 警告 | 重大 | 頻度 |
|---------|-----------|--------|------|------|------|
| **認証** | 認証レイテンシ | < 200ms | > 300ms | > 500ms | 1分 |
| | 認証成功率 | > 98% | < 98% | < 95% | 1分 |
| **SOC** | イベント処理遅延 | < 50ms | > 100ms | > 500ms | 30s |
| | キュー長 | < 1000 | > 5000 | > 10000 | 30s |
| | 脅威検出 | < 15分 | > 20分 | > 30分 | 5分 |
| **インフラ** | CPU使用率 | < 60% | > 75% | > 85% | 1分 |
| | メモリ使用率 | < 70% | > 80% | > 90% | 1分 |
| | ディスク使用率 | < 75% | > 85% | > 95% | 5分 |

### Prometheus アラート設定例

```yaml
groups:
  - name: authentication_alerts
    rules:
      - alert: HighAuthLatency
        expr: auth_latency_p99 > 500
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Auth latency: {{ $value }}ms"

  - name: soc_alerts
    rules:
      - alert: EventQueueBacklog
        expr: soc_queue_length > 10000
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "SOC queue backlog: {{ $value }} events"

  - name: infrastructure_alerts
    rules:
      - alert: DiskSpaceCritical
        expr: node_disk_available / node_disk_total < 0.05
        for: 10m
        labels:
          severity: critical
```

```
  - 毎日 00:30 (1年保持, S3 Glacier)
```

### 復旧手順 - ノード故障

```bash
# 1. 故障確認
kubectl get nodes
# NAME         STATUS   AGE
# node1        Ready    30d
# node2        NotReady 25d  ← 故障

# 2. Pod 移行待機 (自動)
kubectl get pods -o wide --all-namespaces | grep node2

# 3. ノード排出
kubectl drain node2 --ignore-daemonsets --delete-emptydir-data

# 4. ノード復旧
# (ノード再起動 または 置換)

# 5. 復帰確認
kubectl get nodes
kubectl get pods --all-namespaces | grep -c "Running"
```

### 復旧手順 - DB 故障

```bash
# 1. スタンバイへのフェイルオーバー (自動化: Patroni)
patroni ctl switchover postgres

# 2. 接続確認
psql -h db-failover.internal -c "SELECT version();"

# 3. リプリケーション再構築
patroni ctl reinit cluster postgres

# 4. ヘルスチェック
/scripts/health_check.sh
```

## 👥 ユーザー管理

### 新規ユーザー作成

```bash
#!/bin/bash
# create_user.sh

USER_EMAIL=$1
USER_NAME=$2

curl -X POST https://api.security-platform.com/v1/auth/register \
  -H "Authorization: Bearer $API_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"username\": \"$USER_EMAIL\",
    \"password\": \"$(openssl rand -base64 12)@Aa1\",
    \"full_name\": \"$USER_NAME\",
    \"mfa_preference\": \"totp\"
  }"
```

### デバイス信頼スコア管理

```
信頼スコア範囲:

90-100: 信頼できる
  ✓ 既知デバイス, 異常なし, MFA 不要

70-89: 部分的信頼
  ⚠ 新規デバイス (初回30日), MFA 要求

50-69: 低信頼
  ⚠ 異常振る舞い, MFA 必須

0-49: 信頼できない
  ✗ 複数異常検知, 追加検証必須
```

## 🚨 セキュリティ事象対応

### インシデント対応フロー

```
検出 (0-5分)
  ↓ [自動アラート/手動報告]
分類 (5-15分)
  ↓ [重大度/影響範囲判定]
対応 (15-30分)
  ↓ [自動対応/手動対応]
復旧 (30-60分)
  ↓ [システム復旧/ユーザー復帰]
分析 (翌営業日)
  ↓ [ポストモーテム/再発防止]
```

### 対応チェックリスト

```
【検出】
[ ] アラート確認 / ユーザー報告受け取り
[ ] 重大度判定 (Critical/High/Medium/Low)
[ ] オンコール召集

【分類】
[ ] 影響範囲確認 (ユーザー数/システム/データ)
[ ] 根本原因推測 (ログ分析)
[ ] SLA 確認 (対応期限)

【対応】
[ ] 自動対応実行 (Block/Isolate/Revoke)
[ ] ユーザー通知発行
[ ] 経営層定期報告 (30分ごと)

【復旧】
[ ] 影響排除確認
[ ] ヘルスチェック実施
[ ] ユーザー復帰確認

【分析】
[ ] ポストモーテム (翌営業日)
[ ] 再発防止対策検討
[ ] ドキュメント更新
```
