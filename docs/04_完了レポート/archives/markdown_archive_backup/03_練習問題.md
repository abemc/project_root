# Attentionメカニズム - 練習問題とクイズ

## レベル1: 基本概念の理解

### Q1.1: Attention の3要素
**問題**: Attention メカニズムの3つの要素である Query、Key、Value の役割を説明してください。

<details>
<summary>解答</summary>

- **Query (Q)**: 現在のトークンが「何に注目すべきか」を表現。デコーダーの隠れ状態など。
- **Key (K)**: 他のトークンの「特徴」を表現。エンコーダーの隠れ状態など。
- **Value (V)**: 他のトークンの「情報」を表現。実際に加重平均される値。

例: 翻訳タスク
```
Query: デコーダーが現在生成している単語
Key: エンコーダーの各単語
Value: エンコーダーの各単語の表現

Cross-Attention により、
Query が Key に基づいて Value の中から関連情報を抽出
```

</details>

### Q1.2: softmax の役割
**問題**: Attention スコア計算で softmax を使用する理由を説明してください。

<details>
<summary>解答</summary>

softmax の役割:
1. **確率分布への変換**: スコアを 0-1 の確率に変換
2. **合計=1**: 全ウェイトの合計が必ず 1 になる（加重平均の実現）
3. **微分可能性**: ニューラルネットワークの学習に適している

例:
```
QK^T の出力: [5.2, 3.1, 1.8]  ← スコア
softmax 適用: [0.68, 0.26, 0.06]  ← 確率分布

68% を最初のトークンに、
26% を2番目、6% を3番目に
注目を配分
```

</details>

### Q1.3: √d_k でスケーリングする理由
**問題**: Attention スコア計算で $\sqrt{d_k}$ で除算する理由を説明してください。

<details>
<summary>解答</summary>

**勾配爆発を防ぐため**

理由:
```
QK^T の値は、ベクトル次元 d_k が大きいほど大きくなる傾向

例:
- d_k = 4 のとき: QK^T の値が通常 [-10, 10] の範囲
- d_k = 512 のとき: QK^T の値が [-5000, 5000] の範囲

softmax の入力値が大きいと:
- softmax の勾配が非常に小さくなる (< 0.001)
- 勾配消失問題が発生
- 学習が進まない

解決法: √d_k で正規化
- √512 ≈ 22.6 で除算
- 勾配が安定した値に
```

数学的説明:
```
E[QK^T] = E[Q] · E[K^T]

Q, K の要素が独立同分布 (iid) の場合:
E[(QK^T)_i,j] = 0
Var[(QK^T)_i,j] = d_k

√d_k で正規化すると:
Var[QK^T / √d_k] = 1 ← 安定した分布
```

</details>

## レベル2: 実装の理解

### Q2.1: Multi-Head Attention の計算
**問題**: `d_model = 512`, `num_heads = 8` の場合、各ヘッドの次元 `d_k` は？

<details>
<summary>解答</summary>

```
d_k = d_model / num_heads
d_k = 512 / 8 = 64

各ヘッドが 64 次元ベクトルで計算
8つのヘッドの出力を連結すると 512 次元に戻る
```

利点:
- 異なる部分空間で計算 (複数の表現学習)
- 計算複雑さはほぼ変わらない

</details>

### Q2.2: Multi-Head Attention の計算複雑度
**問題**: Input の形状が `(batch_size=2, seq_len=100, d_model=512)` の場合、Multi-Head Attention の計算複雑度は？

<details>
<summary>解答</summary>

```
計算ステップ:

1. 線形変換 (Q, K, V): O(d_model^2) × 3
   = O(512^2) × 3 = O(786,432) per token

2. QK^T: O(seq_len^2 · d_k)
   = O(100^2 · 64) = O(640,000)
   per head, num_heads 分
   = O(640,000 × 8) = O(5,120,000)

3. softmax + Value乗算: O(seq_len^2 · d_k)
   = O(640,000 × 8) = O(5,120,000)

4. 出力層: O(d_model^2)
   = O(262,144)

総計 (単一サンプル):
O(11,788,576) ≈ O(1.2 × 10^7)

batch_size = 2:
O(2.4 × 10^7)

支配的項: O(seq_len^2 · d_model)
= O(100^2 × 512) = O(5,120,000)
```

</details>

### Q2.3: Attention ウェイトの形状
**問題**: 以下のパラメータで Multi-Head Attention を適用した場合、Attention ウェイトの形状は？
- Input: `(batch_size=2, seq_len=10, d_model=64)`
- num_heads: 4

<details>
<summary>解答</summary>

```
Attention ウェイト = softmax(QK^T / √d_k)

形状計算:
Q: (batch=2, seq_len=10, d_model=64)
  → 分割: (batch=2, num_heads=4, seq_len=10, d_k=16)

K: (batch=2, seq_len=10, d_model=64)
  → 分割: (batch=2, num_heads=4, seq_len=10, d_k=16)

QK^T: (batch=2, num_heads=4, seq_len=10, seq_len=10)

したがって、Attention ウェイト形状:
(batch_size=2, num_heads=4, seq_len=10, seq_len=10)

性質:
- 各ヘッドで独立に 10×10 の注目行列
- 各行の合計 = 1 (確率分布)
```

</details>

## レベル3: 応用と最適化

### Q3.1: Efficient Attention との比較
**問題**: 標準的な Attention と Linear Attention の計算複雑度の違いを説明してください。

<details>
<summary>解答</summary>

| 方式 | 計算複雑度 | メモリ | 用途 |
|:------|:------------------------:|:-----------:|:--------------------------|
| Standard Attention | O(n^2 · d) | O(n^2) | 短～中程度のシーケンス |
| Linear Attention | O(n · d) | O(n) | 長いシーケンス |
| Sparse Attention | O(n log n · d) | O(n log n) | 非常に長いシーケンス |

例: n = 4096 (長いテキスト)
```
Standard:
  計算: O(4096^2 × 512) = O(8.6 × 10^9)
  メモリ: O(16.8M) 注目行列

Linear:
  計算: O(4096 × 512) = O(2.1 × 10^6)
  メモリ: O(4K)

Linear Attention が 4000倍高速、メモリ 4000倍削減！
```

トレードオフ:
- Linear Attention は「正確さ」が若干低下
- ほとんどの タスクではほぼ同等の性能

</details>

### Q3.2: Positional Encoding の必要性
**問題**: Attention メカニズムでは位置情報が失われます。これを解決する方法と理由を説明してください。

<details>
<summary>解答</summary>

**Attention の問題: 位置情報の喪失**

```
"猫が犬を見た" vs "犬が猫を見た"

Q, K, V は単語の順序に依存しない
→ 両文の表現が同じになる可能性

解決法: Positional Encoding (位置符号化)
```

**方法1: Absolute Positional Encoding**
```
各位置に固有のベクトルを割り当て
位置 0: [1, 0, 0, ...]
位置 1: [0, 1, 0, ...]
位置 2: [0, 0, 1, ...]

問題: 長いシーケンスに対応困難
```

**方法2: Sinusoidal Positional Encoding**
```
PE(pos, 2i) = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

利点:
- 任意長のシーケンスに対応
- 相対位置を自動学習
- 周期的なパターン
```

**方法3: Rotary Position Embedding (RoPE)**
```
2D の回転行列で相対位置を表現

位置差 k を角度として回転:
θ = k × base_angle

利点:
- 外挿性能が優れている
- 計算効率が良い
- 最新モデルで採用 (LLaMA, ChatGPT等)
```

</details>

### Q3.3: Multi-Query Attention (MQA) の最適化
**問題**: Multi-Query Attention が推論速度を大幅に改善する理由を説明してください。

<details>
<summary>解答</summary>

**背景: KVキャッシュ問題**

標準 Multi-Head Attention:
```
Query: (batch, num_heads, seq_len, d_k)
Key: (batch, num_heads, seq_len, d_k)
Value: (batch, num_heads, seq_len, d_k)

推論時に前の Token の K, V をキャッシュ:
KVキャッシュサイズ = 2 × seq_len × num_heads × d_k
                   = 2 × 2048 × 32 × 128
                   = 16.8 MB (単一バッチ)

バッチサイズ 32:
                   = 538 MB (MemoryBound!)
```

**Multi-Query Attention の解決**

```
Query: (batch, num_heads=32, seq_len, d_k=16)
Key: (batch, num_heads=1, seq_len, d_k=512)
Value: (batch, num_heads=1, seq_len, d_k=512)

計算:
各 Query ヘッドが共通の K, V を使用

KVキャッシュサイズ = 2 × seq_len × 1 × 512
                   = 2 × 2048 × 512
                   = 2.1 MB (8倍削減!)

バッチサイズ 32:
                   = 67 MB (Memory Efficient!)
```

効果:
- メモリ帯域幅が主なボトルネック → 緩和
- 推論スループット: 8倍改善
- 精度低下: ほぼなし (< 1%)

実装:
```python
class MultiQueryAttention:
    def __init__(self, num_heads: int, d_model: int):
        self.num_query_heads = num_heads
        self.num_kv_heads = 1  ← ここ!
        
        self.wq = Linear(d_model, num_heads * d_k)
        self.wk = Linear(d_model, d_k)  ← 8倍小さい!
        self.wv = Linear(d_model, d_k)  ← 8倍小さい!
```

</details>

### C4.2: Attention スコアのデバッグ
**チャレンジ**: Attention ウェイトの合計が1でない場合の原因を特定してください。

```python
def debug_attention(Q, K, V):
    scores = torch.matmul(Q, K.transpose(-2, -1))
    # バグ1: スケーリング忘れ
    # scores = scores / math.sqrt(d_k)  ← コメント化
    
    # バグ2: softmax の次元ミス
    attention_weights = torch.softmax(scores, dim=1)  ← dim=-1 が正しい
    
    # バグ3: Value が異なる形状
    output = torch.matmul(attention_weights, V.transpose(-2, -1))  ← transpose は不要
    
    # 検査
    print(f"Weight sum: {attention_weights.sum(dim=-1)}")
    return output, attention_weights
```

<details>
<summary>バグと修正</summary>

**バグ1: スケーリング忘れ**
```
症状: ウェイトがほぼ 0 と 1 に二分化
原因: softmax の入力が大きすぎて勾配消失
修正: scores / sqrt(d_k) を追加
```

**バグ2: softmax の次元ミス**
```
症状: 各行の合計が 1 でなくなる
原因: dim=1 で softmax → 行ごとでなく列ごとに正規化
修正: dim=-1 に変更 (最後の次元で正規化)
```

**バグ3: Value の転置ミス**
```
症状: 形状のミスマッチでランタイムエラー
原因: V.transpose は不要
修正: torch.matmul(attention_weights, V) に変更
```

正しいコード:
```python
def debug_attention(Q, K, V):
    d_k = Q.size(-1)
    scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(d_k)
    attention_weights = torch.softmax(scores, dim=-1)
    output = torch.matmul(attention_weights, V)
    
    # 検査: 各行の合計が 1
    weight_sums = attention_weights.sum(dim=-1)
    assert torch.allclose(weight_sums, torch.ones_like(weight_sums))
    
    return output, attention_weights
```

</details>

## 解答例

### Q1.1 の実装例

```python
import torch
import torch.nn as nn

class SimpleAttention(nn.Module):
    def forward(self, Q, K, V):
        # Query, Key, Value の役割を実装
        
        # 1. Query が Key から関連度を計算
        scores = torch.matmul(Q, K.transpose(-2, -1))
        
        # 2. softmax で重みに変換
        weights = torch.softmax(scores, dim=-1)
        
        # 3. Value を加重平均
        output = torch.matmul(weights, V)
        
        return output, weights

# デモ
Q = torch.randn(1, 10, 64)  # 10個のクエリ
K = torch.randn(1, 10, 64)  # 10個のキー
V = torch.randn(1, 10, 64)  # 10個のバリュー

attn = SimpleAttention()
output, weights = attn(Q, K, V)

print(f"Output shape: {output.shape}")  # (1, 10, 64)
print(f"Weights shape: {weights.shape}")  # (1, 10, 10)
print(f"Weight sum: {weights.sum(dim=-1)}")  # すべて1
```
